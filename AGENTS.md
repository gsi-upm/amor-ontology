# AMOR ontology release instructions

## Mission

This repository is being prepared as the reproducible software and ontology
release accompanying the AMOR ontology paper. Work in this repository should
currently focus on the following release tasks:

1. publish executable competency-question queries;
2. repair and extend the automated tests;
3. make documentation generation and the web release traceable to one Git
   commit;
4. prepare, but do not create without explicit approval, release `v1.0.0`.

Do not broaden the task into redesigning the ontology, the experimental API,
or the paper unless the user explicitly requests it.

## Sources of truth and module boundaries

Treat the Turtle files under `src/` as the normative, version-controlled RDF
sources. The publication under `gsi.upm.es/ontologies` is the intended public
web release and must be generated from the same commit that is eventually
tagged.

The architecture is modular:

- `src/amor/amor.ttl`: AMOR core ontology.
- `src/bhv/bhv.ttl`: BHV SKOS taxonomy.
- `src/mft/mft.ttl`: MFT SKOS taxonomy.
- `src/amor/models/bhv/amor-bhv.ttl`: connection between BHV concepts and the
  AMOR core.
- `src/amor/models/mft/amor-mft.ttl`: connection between MFT concepts and the
  AMOR core, including MFT-specific polarity.
- `src/amor/examples/amor-examples.ttl`: compact examples used for competency
  questions.
- `src/amor/experiments/`: a separate experimental module. It is not part of
  the AMOR-core competency-question evaluation.
- `src/amor/datasets/*.ttl`: article-level news datasets that may be used for
  illustrative instantiation validation.

Preserve the published ontology IRIs exactly as declared in the sources. In
particular, do not rewrite `http` to `https`, remove `www`, or remove `/ns#`
from vocabulary namespaces merely because the human-facing landing pages use
different URLs.

There is no formal AMOR-to-ValueNet mapping. AMOR and ValueNet cover some of
the same theoretical frameworks, but no RDF/OWL/SKOS alignment between them
may be invented or implied.

## Decisions already made

The following decisions are settled and must not be reopened without new
evidence or an explicit user request:

- `mft:Fairness` is retained and included in the consolidated
  `amor-mft:MoralFoundationsTheoryModel`.
- The MFT model is described as a consolidated category set, not as a strict
  reproduction of only the 2023 formulation.
- The labels of `mft:Virtue` and `mft:Vice` must match their IRIs.
- BHV concepts use the namespace
  `http://www.gsi.upm.es/ontologies/bhv/ns#`.
- In `moralAnnotation23`, `mft:Authority` is the value of
  `amor:hasMoralValueCategory`; the annotated news item is the value of
  `oa:hasTarget`.
- The JSON files produced for the experimental API are intentionally framed,
  application-oriented views for direct JSON traversal. They are not
  normative RDF serializations and are not expected to be graph-isomorphic to
  the corresponding Turtle files.
- The API-oriented JSON views and their converters are outside the paper's
  evaluation. Do not change them to enforce Turtle/JSON-LD equivalence as part
  of this release work.
- The unreproducible coverage table from the draft paper will not be rescued
  by inventing a new calculation. Repository validation is based on
  competency questions and RDF instantiation examples.

Some accepted ontology/example fixes may initially exist only in the user's
working tree. Inspect the actual diff before editing. Verify those fixes; do
not overwrite or duplicate them.

## Task 1: competency questions

Create one standalone SPARQL query per competency question, preferably under:

```text
evaluation/
  README.md
  competency-questions/
    CQ-C1.rq
    CQ-C2.rq
    CQ-C3.rq
    CQ-C4.rq
    CQ-C5.rq
    CQ-C6.rq
    CQ-C7.rq
    CQ-C8.rq
    CQ-C9.rq
    CQ-M1.rq
```

Each `.rq` file must include every prefix it uses so that it can be executed
without concatenating `src/prefixes.sparql` or relying on application state.

Use these competency questions:

| ID | Competency question | Principal terms |
|---|---|---|
| CQ-C1 | What resource was analysed in a moral-value analysis? | `amor:MoralValueAnalysis`, `amor:analysed` |
| CQ-C2 | Which moral-value annotations were generated by an analysis? | `prov:generated`, `amor:MoralValueAnnotation` |
| CQ-C3 | What resource is targeted by a moral-value annotation? | `oa:hasTarget` |
| CQ-C4 | Which moral-value model was used in an analysis? | `amor:usedMoralValueModel` |
| CQ-C5 | Which moral-value category was assigned by an annotation? | `amor:hasMoralValueCategory` |
| CQ-C6 | Which categories are included in a moral-value model? | `amor:MoralValueModel`, `amor:hasMoralValueCategory` |
| CQ-C7 | Which agent was responsible for an analysis? | `prov:wasAssociatedWith` |
| CQ-C8 | What confidence score is associated with an annotation? | `itsrdf:taConfidence` |
| CQ-C9 | Which ML model was used in an analysis? | `amor:usedMLModel`, `mls:Model` |
| CQ-M1 | What MFT-specific polarity and intensity are associated with an annotation? | `amor-mft:hasPolarity`, `amor-mft:hasPolarityIntensity` |

The evaluation graph is the union of:

```text
src/amor/amor.ttl
src/bhv/bhv.ttl
src/mft/mft.ttl
src/amor/models/bhv/amor-bhv.ttl
src/amor/models/mft/amor-mft.ttl
src/amor/examples/amor-examples.ttl
```

Do not load the experiments ontology to make a core query pass. Document the
graph composition and execution command in `evaluation/README.md`.

On the corrected compact example, the current expected row counts are 4 for
CQ-C1, C2, C3, C4, C5, C7, and C8; 2 for CQ-C9; and 3 for CQ-M1. For CQ-C6,
prefer semantic assertions over one brittle total: verify the five BHV model
individuals, the consolidated MFT model, `bhv:Conservation` in
`amor-bhv:BHVModel-level3`, and `mft:Fairness` in the MFT model. Assert key
bindings as well as counts so that a duplicated or incorrect row cannot make
a test pass accidentally.

## Task 2: automated tests

The current `tests/test_experiments.py` is stale and is not evidence for the
paper's core evaluation. Known issues include:

- `RecommendationStrategy` was renamed/replaced by
  `RecommendationCriterion`;
- it references example files in paths that no longer exist;
- the dataset relation is `amor-exp:usesDataset`, not `hasDataset`;
- tests that query `amor-exp:hasParameter` directly ignore its subproperties
  and perform no RDFS inference;
- `unittest.skip` is used without an explanatory reason;
- its JSON tests concern an old API shape and must not be presented as RDF
  equivalence tests.

Repair or remove obsolete tests intentionally. Do not leave unexplained
skips merely to make CI green.

Add focused tests, using clear names such as:

```text
tests/test_rdf_syntax.py
tests/test_competency_questions.py
tests/test_models.py
```

At minimum, test that:

1. every normative Turtle source parses locally;
2. every competency-question file parses and returns the expected bindings;
3. the evaluation uses exactly the documented graph inputs;
4. `mft:Fairness` belongs to the consolidated MFT model;
5. `mft:Virtue` and `mft:Vice` have the correct preferred/display labels;
6. the BHV example resolves to `.../bhv/ns#Conservation`;
7. `moralAnnotation23` assigns `mft:Authority` through
   `amor:hasMoralValueCategory` and has only the news resource as its target.

Tests must run offline. Replace remote JSON-LD contexts with local fixtures
when a JSON test is genuinely retained. Do not make CI depend on the live GSI
website.

Use the locked environment:

```bash
UV_CACHE_DIR=/tmp/uv-cache uv run --frozen pytest -q
```

Add a small GitHub Actions workflow that runs the same command on pushes and
pull requests. Pin action major versions and the Python version; do not add
deployment credentials or a production deployment job.

## Task 3: documentation, web publication, and release

The existing WIDOCO scripts generate documentation but do not document or
perform deployment to `gsi.upm.es`. They currently use the mutable container
tag `ghcr.io/dgarijo/widoco:latest`. Make documentation generation
reproducible by pinning a specific WIDOCO version or image digest and recording
it in the README. Do not guess a version: resolve and record the version that
is actually used for the release.

Generated WIDOCO graphs may contain additional declarations or harmless
serialization changes. Do not require byte identity between a source Turtle
file and WIDOCO's documentation output. Instead:

- retain the source Turtle files as normative;
- record hashes for normative release artefacts;
- parse generated RDF outputs;
- verify ontology IRIs, version metadata, and the key terms affected by this
  release;
- ensure the human-readable documentation reflects the corrected source.

Before changing publication automation, determine how the GSI server is
currently updated: manual copy, `scp`, `rsync`, a separate web repository, or
another mechanism. Ask the user if this cannot be established from repository
evidence. Do not attempt to access or mutate the production server without
explicit authorization and a confirmed deployment procedure.

Recommended release order:

1. inspect and preserve the user's ontology/example fixes;
2. add queries, tests, CI, and reproducible documentation scripts;
3. run the complete offline test suite;
4. regenerate and commit documentation from the same source commit;
5. publish the web artefacts from that commit using the confirmed procedure;
6. retrieve and parse the public representations and verify the affected
   terms;
7. only then create annotated tag `v1.0.0` and the corresponding GitHub
   Release.

Creating a Git tag or GitHub Release is an external, consequential action.
Prepare the command or release notes, but do not create either without an
explicit user instruction after the final commit and web publication have
been verified.

## Acceptance criteria

Release work is complete only when all of the following are true:

- the accepted ontology and example fixes are present in the final diff;
- all normative Turtle files parse;
- all ten competency-question queries run against the documented graph;
- the tests assert the expected semantic bindings, not only non-empty output;
- `pytest` completes without failures or unexplained skips;
- GitHub Actions runs the same offline validation successfully;
- documentation generation uses a pinned WIDOCO version or digest;
- generated RDF documentation parses and contains the corrected terms;
- the public web artefacts have been verified against the release commit;
- the final commit is the commit tagged `v1.0.0`;
- the paper/release documentation cites that tag or its archived identifier.

## Working practices

- Start by inspecting `git status`, the current branch, and the relevant
  diffs. Existing changes belong to the user.
- Work on a dedicated branch and use a pull request unless the user explicitly
  asks for a different workflow. Do not push directly to `main`.
- Use `rg` for repository searches and `apply_patch` for manual edits.
- Keep changes scoped. Do not rename public ontology terms or mint new IRIs to
  make tests easier.
- Do not silently add OWL axioms, mappings, categories, datasets, or evaluation
  results.
- Distinguish confirmed repository evidence, reasonable inference, pending
  verification, and unsupported claims in progress reports.
- Report the exact commands executed and their results. If a check was
  simulated against an uncommitted correction, say so explicitly.
- The paper source may be supplied separately from this repository. Do not edit
  the manuscript as part of repository release work unless specifically asked.
